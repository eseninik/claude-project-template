# Universal Parallelization Proposal

**Проблема:** Текущее AUTO-CHECK требует явный файл плана. Неявные планы не обнаруживаются.

**Цель:** Универсальное определение возможности параллелизации без зависимости от формата.

---

## Проблема

### Сценарий пользователя

> "Иногда я просто прошу сделать автоматически, агент создаёт небольшой план и выполняет последовательно"

**Что происходит:**
1. Пользователь: "Добавь JWT аутентификацию"
2. Агент создаёт план в памяти:
   - Создать middleware для проверки токена
   - Добавить endpoint для логина
   - Написать тесты
   - Обновить документацию
3. Эти задачи НЕЗАВИСИМЫЕ (можно параллельно)
4. НО агент работает последовательно (нет файла плана)

**Текущее AUTO-CHECK:**
```
AUTO-CHECK: Есть work/*/tasks/*.md?
  └─ НЕТ → продолжить обычным workflow → последовательно
```

**Результат:** Потеря производительности.

---

## Решение: Контекстный анализ

### Новое AUTO-CHECK (v2)

```
AUTO-CHECK: Есть возможность параллелизации?

ЭТАП 1: Проверка явного плана
  A. Есть work/*/tasks/*.md с 3+ задачами?
     └─ ДА → subagent-driven-development

  B. Есть plan.md с этапами?
     └─ ДА → Предложить конвертацию

ЭТАП 2: Анализ контекста задачи (НОВОЕ)
  C. Задача требует 3+ независимых изменений?
     └─ Анализ:
        - Парсинг описания задачи
        - Выявление подзадач
        - Проверка независимости

     └─ ДА → ПРЕДЛОЖИТЬ:
        "Вижу 3+ независимые подзадачи:
         1. [подзадача 1]
         2. [подзадача 2]
         3. [подзадача 3]

         Варианты:
         1. Создать tasks/*.md и выполнить параллельно (быстрее)
         2. Работать последовательно
         3. Спросить у пользователя детали"

ЭТАП 3: Fallback
  D. Нет признаков параллелизации
     └─ Продолжить обычным workflow
```

---

## Алгоритм контекстного анализа

### Триггеры для "3+ независимых изменений"

**Паттерны в тексте задачи:**

1. **Списки действий:**
   ```
   "Нужно:
    - Добавить X
    - Создать Y
    - Обновить Z"
   ```
   → Обнаружить: 3+ пункта

2. **Ключевые слова:**
   ```
   "Добавь JWT аутентификацию"
   ```
   → Знаю что аутентификация = middleware + endpoints + tests + docs
   → 4 независимые задачи

3. **Множественные файлы:**
   ```
   "Обнови src/user.py, src/api.py, src/auth.py"
   ```
   → 3 файла = потенциально 3 задачи

4. **Фразы-индикаторы:**
   - "и также", "кроме того", "плюс", "еще"
   - "сначала X, потом Y, затем Z"
   - "для каждого из", "все компоненты"

### Проверка независимости

**Для каждой пары подзадач:**
```
Задача A и Задача B независимые, если:
- Разные файлы ИЛИ
- Один файл, но разные функции/классы ИЛИ
- Одна функция, но разные аспекты (можно в Worktree Mode)
```

**Зависимые задачи:**
```
Задача B зависит от A, если:
- B использует результат A в том же файле/функции
- B требует чтобы A выполнилась первой
```

## Пример работы

### Запрос пользователя

```
"Добавь систему уведомлений для пользователей"
```

### Анализ агента

**ЭТАП 1: Явный план?**
- Нет tasks/*.md → НЕТ
- Нет plan.md → НЕТ

**ЭТАП 2: Контекстный анализ**

Парсинг задачи:
```
"Система уведомлений" → Знаю что это включает:
1. Модель для уведомлений (models/notification.py)
2. API endpoints (api/notifications.py)
3. Email-сервис (services/email.py)
4. Webhook-интеграция (webhooks/notifications.py)
5. Тесты (tests/test_notifications.py)
```

Проверка независимости:
```
1 ↔ 2: Разные файлы → НЕЗАВИСИМЫЕ ✓
1 ↔ 3: Разные файлы → НЕЗАВИСИМЫЕ ✓
2 ↔ 3: Разные файлы → НЕЗАВИСИМЫЕ ✓
...
```

**Вывод:** 5 независимых задач

### Предложение пользователю

```
Вижу что задача включает 5 независимых компонентов:
1. Модель Notification
2. API endpoints
3. Email сервис
4. Webhook integration
5. Тесты

Варианты:
1. Создать work/notifications/tasks/*.md и выполнить параллельно (рекомендуется)
   → Этапы 1-3 параллельно, затем 4, затем 5
   → Сэкономит ~60% времени

2. Работать последовательно
   → Проще, но медленнее

3. Дать мне больше деталей для точного плана

Какой вариант предпочитаете?
```

## Преимущества

**Для пользователя:**
- ✅ Не нужно явно создавать plan.md
- ✅ Агент сам видит возможность оптимизации
- ✅ Пользователь выбирает (не навязывание)

**Для производительности:**
- ✅ Больше случаев параллелизации
- ✅ Меньше упущенных возможностей

## Риски и ограничения

### Риск 1: Ложные срабатывания

**Проблема:** Агент видит 3 задачи, но они НЕ независимые

**Митигация:**
- Проверка зависимостей (файлы, функции)
- Спрашивать пользователя при неуверенности
- Консервативный подход: при сомнении → sequential

### Риск 2: Пропуск возможностей

**Проблема:** Агент НЕ видит что задачи можно распараллелить

**Митигация:**
- База знаний паттернов ("аутентификация" → middleware + endpoints + tests)
- Learning from user feedback
- Fallback: всегда можно создать tasks/*.md вручную

### Риск 3: Навязчивость

**Проблема:** Агент спрашивает про параллелизацию даже когда не нужно

**Митигация:**
- Порог: только если 3+ задачи И уверенность > 80%
- Короткое предложение (не длинное объяснение)
- Вариант "больше не спрашивать для этой сессии"

## Реализация

### Фаза 1: База знаний

Создать `.claude/knowledge/parallelization-patterns.md`:

```markdown
# Паттерны задач для параллелизации

## Аутентификация
- Middleware для проверки токена
- Endpoint для логина
- Endpoint для регистрации
- Тесты
- Документация

Независимость: ДА (разные файлы)
Waves: [middleware, login, register, docs] → [tests]

## CRUD для модели
- Модель (models/X.py)
- API endpoints (api/X.py)
- Валидация (validators/X.py)
- Тесты (tests/test_X.py)

Независимость: ДА (разные файлы)
Waves: [model, api, validators] → [tests]

## ...
```

### Фаза 2: Обновить AUTO-CHECK

Добавить в CLAUDE.md:

```markdown
## Dynamic Skill Selection

```
1. СТОП
2. AUTO-CHECK: Есть возможность параллелизации?

   A. План в формате work/*/tasks/*.md?
      └─ ДА → subagent-driven-development

   B. План в другом формате (plan.md)?
      └─ ДА → Предложить конвертацию

   C. Контекстный анализ задачи (NEW)
      └─ Загрузить .claude/knowledge/parallelization-patterns.md
      └─ Проверить паттерны в описании задачи
      └─ Если 3+ подзадачи → Предложить варианты

   D. Нет признаков параллелизации
      └─ Продолжить обычным workflow
```

### Фаза 3: Skill для анализа

Создать `.claude/skills/task-decomposition/SKILL.md`:

```markdown
# Task Decomposition Skill

Анализирует задачу и выявляет возможность разбиения на параллельные подзадачи.

## Алгоритм

1. Парсинг описания задачи
2. Выявление подзадач (паттерны, списки)
3. Проверка независимости
4. Оценка файловых конфликтов
5. Рекомендация (parallel / sequential)

## Используется

- AUTO-CHECK вариант C (контекстный анализ)
- По запросу: "Можно ли распараллелить эту задачу?"
```
