# v2.0 Testing Examples

**Тестирование универсального определения параллелизации.**

## Пример 1: Системная фича (высокая уверенность)

### Запрос пользователя

```
"Добавь систему уведомлений"
```

### Процесс агента (AUTO-CHECK вариант C)

**Шаг 1: Парсинг**
```
Ключевое слово: "уведомлен*"
Проверка паттернов: cat .claude/knowledge/parallelization-patterns.md
Совпадение: Паттерн "Notifications"
```

**Шаг 2: Извлечение подзадач из паттерна**
```
1. Модель Notification (models/notification.py)
2. API endpoints (api/notifications.py)
3. Email сервис (services/email.py)
4. Push notification (services/push.py)
5. Webhook integration (webhooks/notifications.py)
6. Тесты (tests/test_notifications.py)
```

**Шаг 3: Проверка независимости**
```
1 ↔ 2: Разные файлы → INDEPENDENT
1 ↔ 3: Разные файлы → INDEPENDENT
2 ↔ 3: Разные файлы → INDEPENDENT
...
Тесты зависят от всех → DEPENDENT
```

**Шаг 4: Построение waves**
```
Wave 1: [1, 2, 3, 4, 5] (5 задач параллельно)
Wave 2: [6] (тесты зависят от всех)
```

**Шаг 5: Уверенность**
```
Паттерн совпал: +30%
Разные файлы: +10%
Чёткие зависимости: +10%
Итого: 90% → ВЫСОКАЯ
```

**Шаг 6: Checkpoint**
```
Situation: Добавление системы уведомлений
Plan format: implicit
Plan tasks count: 6
Parallelization analysis: detected
  Subtasks:
    1. Модель Notification (models/)
    2. API endpoints (api/)
    3. Email сервис (services/)
    4. Push notification (services/)
    5. Webhook integration (webhooks/)
    6. Тесты (tests/)
  Waves:
    - Wave 1: [1, 2, 3, 4, 5] (параллельно)
    - Wave 2: [6] (после всех)
  Confidence: high (90%)
Skills: task-decomposition
```

**Шаг 7: Предложение пользователю**
```
Вижу что задачу можно разбить на 6 подзадач:

Wave analysis:
- Wave 1: [1,2,3,4,5] (5 задач параллельно)
- Wave 2: [6] (тесты после всех)

Варианты:
1. Создать work/notifications/tasks/*.md и выполнить параллельно (рекомендуется)
   → Сэкономит ~80% времени (5 задач одновременно)
2. Работать последовательно
3. Дать больше деталей для уточнения плана

Какой вариант?
```

**Ожидаемое:** Пользователь выберет вариант 1 → агент создаст tasks/*.md → subagent-driven-development

## Пример 2: Список задач (средняя уверенность)

### Запрос пользователя

```
"Нужно:
- Добавить логирование в endpoints
- Обновить README
- Написать smoke tests"
```

### Процесс агента

**Шаг 1: Парсинг**
```
Структура: Список из 3 пунктов
Паттерн: Нет прямого совпадения
Извлечение из списка
```

**Шаг 2: Подзадачи**
```
1. Добавить логирование (api/*.py)
2. Обновить README (README.md)
3. Написать smoke tests (tests/smoke/)
```

**Шаг 3: Проверка независимости**
```
1 ↔ 2: Разные файлы → INDEPENDENT
1 ↔ 3: Логирование может влиять на тесты → UNCERTAIN
2 ↔ 3: Разные файлы → INDEPENDENT
```

**Шаг 4: Waves (предположительно)**
```
Вариант A: Wave 1: [1,2,3] (если тесты независимые)
Вариант B: Wave 1: [1,2], Wave 2: [3] (если тесты зависят от логирования)
```

**Шаг 5: Уверенность**
```
Список: +20%
Разные файлы: +10%
Неуверенность в зависимостях: -20%
Итого: 60% → СРЕДНЯЯ
```

**Шаг 6: Checkpoint**
```
Situation: Добавление логирования, обновление README, smoke tests
Plan format: implicit
Plan tasks count: 3
Parallelization analysis: detected
  Subtasks:
    1. Логирование (api/*.py)
    2. README (README.md)
    3. Smoke tests (tests/smoke/)
  Waves: uncertain (зависят от зависимости тестов)
  Confidence: medium (60%)
Skills: task-decomposition
```

**Шаг 7: Предложение пользователю**
```
Похоже задачу можно разбить на 3 подзадачи, но есть вопрос о зависимостях.

Подзадачи:
1. Логирование в endpoints
2. Обновление README
3. Smoke tests

Вопрос: Smoke tests зависят от логирования, или это независимая задача?

Варианты:
1. Если независимые → все 3 параллельно
2. Если тесты зависят → логирование + README параллельно, затем тесты
3. Работать последовательно

Какой случай у вас?
```

**Ожидаемое:** Пользователь уточнит зависимости → агент создаст tasks/*.md с правильными depends_on

## Пример 3: Неопределённая задача (низкая уверенность)

### Запрос пользователя

```
"Улучши производительность приложения"
```

### Процесс агента

**Шаг 1: Парсинг**
```
Ключевое слово: "производительност*"
Паттерн: Performance Optimization (частичное совпадение)
НО: не ясно какие конкретно оптимизации
```

**Шаг 2: Подзадачи (из паттерна, но неточные)**
```
Типовые для оптимизации:
1. Profiling (?)
2. Database optimization (?)
3. Caching (?)
4. Background jobs (?)

НО: Не знаю какие именно нужны
```

**Шаг 3: Уверенность**
```
Паттерн (частичный): +10%
Неясные детали: -40%
Итого: 20% → НИЗКАЯ
```

**Шаг 4: Checkpoint**
```
Situation: Оптимизация производительности
Plan format: none
Plan tasks count: unknown
Parallelization analysis: uncertain
  Subtasks: типовые (profiling, db, cache, bg jobs) НО не подтверждены
  Confidence: low (20%)
Skills: task-decomposition (анализ не дал результата)
```

**Шаг 5: Предложение пользователю**
```
Задача про оптимизацию производительности.

Типовые подзадачи для такой задачи:
1. Profiling для выявления bottlenecks
2. Database optimization (indexes, queries)
3. Caching layer
4. Background jobs

НО мне нужно больше информации:
- Какие части приложения медленные?
- Уже известны bottlenecks или нужен profiling?
- Какие конкретно оптимизации нужны?

Варианты:
1. Дать детали → создам точный план
2. Начать с profiling → определим bottlenecks → затем план
3. Работать последовательно, разбирая по ходу

Что предпочитаете?
```

**Ожидаемое:** Пользователь даст детали ИЛИ выберет вариант 2/3

## Пример 4: Явная последовательность (не параллелить)

### Запрос пользователя

```
"Создай функцию fetch_data(), затем используя её результат создай generate_report()"
```

### Процесс агента

**Шаг 1: Парсинг**
```
Ключевые слова: "используя её результат" → явная зависимость
Подзадачи:
1. fetch_data()
2. generate_report() - ЗАВИСИТ от 1
```

**Шаг 2: Проверка независимости**
```
1 ↔ 2: generate_report использует результат fetch_data → DEPENDENT
```

**Шаг 3: Waves**
```
Wave 1: [1]
Wave 2: [2] (зависит от 1)
```

**Шаг 4: Оценка**
```
Задач: 2 (< 3) → параллелизация не нужна
```

**Шаг 5: Checkpoint**
```
Situation: Создание fetch_data и generate_report
Plan format: none
Plan tasks count: 2
Parallelization analysis: not applicable (< 3 tasks, sequential)
Skills: (обычный workflow, task-decomposition не применялся)
```

**Шаг 6: Действие**
```
Продолжить обычным workflow (вариант C пункт 4 ELSE)
```

**Ожидаемое:** Работа последовательно, НЕ предлагать параллелизацию

## Пример 5: Создание tasks/*.md после согласия

### Контекст

Пользователь выбрал "Вариант 1: Создать tasks/*.md" после Примера 1.

### Процесс агента

**Шаг 1: Генерация структуры**
```bash
mkdir -p work/notifications/tasks/
```

**Шаг 2: Создание task-001.md**
```markdown
---
depends_on: []
files_modified:
  - models/notification.py
---

# Task 001: Модель Notification

## Описание
Создать модель Notification для хранения уведомлений пользователей.

## Детали реализации
- Поля: id, user_id, type, message, read_at, created_at
- Методы: mark_as_read(), to_dict()
- Миграция БД для таблицы notifications

## Критерии готовности
- [ ] Модель создана в models/notification.py
- [ ] Миграция создана и применена
- [ ] Тесты модели написаны
- [ ] Тесты проходят
- [ ] Committed
```

**Шаг 3: task-002.md, task-003.md, ... (аналогично)**

**Шаг 4: Повторный AUTO-CHECK**
```
AUTO-CHECK вариант A: Есть work/notifications/tasks/*.md с 6 задачами
→ subagent-driven-development ОБЯЗАТЕЛЕН
```

**Шаг 5: subagent-driven-development**
```
Wave analysis:
- Wave 1: [001, 002, 003, 004, 005] (5 параллельно)
- Wave 2: [006] (тесты)

Запуск 5 subagents параллельно...
```

**Результат:** Параллельное выполнение, экономия времени

## Выводы

**v2.0 обнаруживает возможность параллелизации даже без явного файла плана.**

**Ключевые улучшения:**
1. ✅ База знаний паттернов распознаёт типовые задачи
2. ✅ Алгоритм анализирует структуру задачи (списки, файлы, связки)
3. ✅ Оценка уверенности предотвращает навязчивость
4. ✅ Предлагает варианты вместо навязывания
5. ✅ Создаёт tasks/*.md по запросу → subagent-driven-development

**Универсальность достигнута:**
- План в tasks/*.md → параллелизация ✓
- План в plan.md → конвертация → параллелизация ✓
- План в памяти агента → анализ → параллелизация ✓
- Нет плана, < 3 задач → sequential ✓
