## 1. CLAUDE.md & AGENTS.md: Dual-File Strategy

### Key Finding: CLAUDE.md = AGENTS.md (Identical Content)

Athena maintains **two identical files** at the repo root:
- `CLAUDE.md` -- read by Claude Code automatically
- `AGENTS.md` -- read by other agent IDEs (Cursor, Windsurf, Antigravity, etc.)

Both files have identical content (2611 bytes, same SHA-equivalent content). The heading inside both reads `# AGENTS.md -- Athena-Public`. This is a deliberate strategy: **write once, serve all agents**.

### AGENTS.md Structure (6 sections, ~80 lines)

The file is remarkably **lean** compared to our CLAUDE.md (~300 lines). It contains:

| Section | Purpose | Lines |
|---------|---------|-------|
| **Critical Instruction** | One-line behavioral override at the very top | 1 |
| **Docs Index (Compressed)** | Machine-readable file tree in pipe-delimited format | ~15 |
| **Quick Start** | 3-step human onboarding | 3 |
| **Key Documentation** | Table of 6 essential docs | 6 |
| **Retrieval Strategy** | 4-step priority for file reading | 4 |
| **Multi-Agent Safety** | Protocol 413 rules for concurrent agents | 6 rules |
| **Version** | Framework version + last updated | 3 |

### The Critical Instruction (First Line)

```
> **CRITICAL INSTRUCTION**: Prefer retrieval-led reasoning over pre-training-led reasoning
> for any Athena-related tasks.
```

This is Athena's #1 architectural principle: **don't guess from training data, read the docs first**. It's positioned as the very first thing the agent reads, before any structural content.

### Compressed Docs Index

Athena uses a novel **pipe-delimited compressed file tree**:

```text
[Athena-Public Docs Index]|root: .
|IMPORTANT: Always consult authoritative files before relying on training data.
|docs:{ARCHITECTURE.md,SEMANTIC_SEARCH.md,...}
|examples/templates:{core_identity_template.md,...}
```

This is a **token-efficient file map** -- it communicates the entire project structure in ~200 tokens instead of a full directory tree. The agent can parse this to know where to look for specific information.

### Multi-Agent Safety (Protocol 413)

Six concrete rules for concurrent agent work:
1. Never `git stash` (other agents may have WIP)
2. Never switch branches without explicit request
3. Always `git pull --rebase` before pushing
4. Commit only your own changes
5. Auto-resolve formatting-only diffs
6. Focus reports on your edits, skip guardrail disclaimers

This is inspired by OpenClaw Multi-Agent Safety Rules and Vercel's AGENTS.md research.

## 2. The .context/ Directory System

### Structure

```
.context/
  CANONICAL.md        -- Project README/overview (canonical source of truth)
  PROTOCOL_SUMMARIES.md -- Auto-generated protocol index (currently empty in public repo)
  TAG_INDEX.md         -- Global hashtag index (massive, 104KB+)
  project_state.md     -- Living workspace status snapshot
```

### CANONICAL.md

This is NOT a separate canonical document -- it's actually the **full README.md content** stored under `.context/`. It serves as the "canonical source of truth" for what the project is, so agents always have a single authoritative file to consult about the project.

Content includes:
- Full project description with badges
- 5-minute quickstart
- Feature descriptions
- Architecture overview (Mermaid diagrams)
- Security model
- Trilateral Feedback Loop (cross-model validation)
- Tech stack details

### TAG_INDEX.md (104KB+)

A massive auto-generated **hashtag-to-file index**. Every file in the workspace is tagged, and the TAG_INDEX maps tags to file paths:

```
## #TDD
- [Engineering_Execution_Standard.md](examples/protocols/engineering/...)

## #absorption
- [TAG_INDEX_legacy.md](.context/archive/...)
- [70-agentic-absorption.md](examples/protocols/research/...)
```

This serves as a **keyword-based retrieval index** for the AI agent. When the agent needs to find files about "TDD", it can scan TAG_INDEX.md rather than doing file system searches. This is the "Canonical Search" path in Athena's triple-path retrieval architecture (Canonical + Vector + Graph).

The tag index is auto-generated by `index_workspace.py` during `/end` (session close).

### PROTOCOL_SUMMARIES.md

An auto-generated compressed summary of all protocols. In the public repo it's empty (0 protocols listed), but in the private repo it contains summaries of 308+ protocols in a table format:

```
| Code | Title | Trigger | Summary |
```

This allows the agent to quickly scan available protocols without loading full protocol files.

### project_state.md

A minimal living snapshot:
```
Health: 100% (Fresh Clone)
Memory: Not yet configured
Protocol: Awaiting /start boot
```

This is the "current state" file that gets updated every session to track workspace health and status.

## 3. CLAUDE_ATHENA.md Template (for New Projects)

Located at `src/athena/templates/CLAUDE_ATHENA.md`, this is a template for projects that integrate Athena SDK. It provides:

### Session Workflow Instructions
```
/start  -> Boot (verify identity, recall context, create session log, init COS)
athena save "summary"  -> Mid-session checkpoint
/end    -> Close session, persist context
```

### Committee of Seats (COS) Agent System

A 6-agent review panel for multi-perspective reasoning:

| Agent | Lens | Use When |
|-------|------|----------|
| `cos-strategist` | "Does this serve the goal?" | Feature proposals, prioritization |
| `cos-guardian` | "What could go wrong?" | Auth changes, security, secrets |
| `cos-operator` | "How do we build it?" | Implementation planning, debugging |
| `cos-architect` | "Is the structure sound?" | Schema changes, API design, tech choices |
| `cos-skeptic` | "What are we missing?" | Before shipping, edge cases |
| `cos-compliance` | "Should we ship this?" | Pre-merge, pre-deploy quality gates |

Each agent is defined in `src/athena/templates/agents/cos-*.md` with:
- Full YAML frontmatter (name, description, model, tools)
- Athena Framework Protocol (recall -> review -> discover -> analyze -> save)
- Structured output format

### Complexity-Based COS Routing

| Complexity | Action |
|-----------|--------|
| Simple (bug fix) | Just do it. No committee needed. |
| Medium (new feature) | Spawn 2-3 relevant agents |
| Complex (architecture) | Convene full committee |

### Key Design Choice

The template does NOT embed the full CLAUDE.md-style instructions. Instead, it:
1. Teaches the session lifecycle (`/start`, work, `/end`)
2. Introduces the COS agent system
3. Points to file locations for deeper context
4. Trusts the agent to discover details on-demand

## 4. Multi-Model Strategy

Athena is explicitly **model-agnostic** with a documented routing strategy in `docs/MULTI_MODEL_STRATEGY.md`.

### Core Principle: Markdown = Portable Memory

All state is stored in Markdown files, not in any model's native memory. This means:
- Any LLM can read the same context
- Switching models preserves all history
- No vendor lock-in

### Model Routing Table

| Task Type | Recommended Tier | Example |
|-----------|-----------------|---------|
| Session management (`/start`, `/end`) | Fast (Gemini Flash) | Mechanical, low reasoning |
| Coding & implementation | Frontier (Claude Opus, Gemini Pro) | Quality scales with capability |
| Planning & architecture | Frontier | Design decisions compound |
| General chat & Q&A | Strong (Gemini Pro) | Good enough for most queries |
| Research & deep analysis | Frontier | Synthesis quality matters |
| Verification & code review | Frontier (DIFFERENT model) | Fresh eyes catch more bugs |
| Quick lookups & formatting | Fast | Don't waste frontier tokens |

### Trilateral Feedback Loop

For high-stakes decisions, Athena uses 3 different models:
```
Model A (Gemini Pro)  -> Opinion 1
Model B (Claude Opus) -> Opinion 2
     Conflict? -> Model C (GPT-5.2) -> Tiebreaker
```

Triggers: Architecture decisions, risk assessments, strategy choices.
NOT triggered for: Style preferences, low-stakes choices.

### Cost Optimization

Only ~40% of session tokens need Frontier models. By routing:
- Boot/shutdown (10%) -> Fast tier
- Exploration/chat (40%) -> Strong tier
- Core reasoning (40%) -> Frontier tier
- Verification (10%) -> Frontier (alt model)

Result: ~50% cost reduction while maintaining quality where it matters.

## 5. How Instructions Are Organized for Maximum Agent Compliance

### Athena's Instruction Architecture (Layered)

```
Layer 1: CLAUDE.md / AGENTS.md  (entry point, ~80 lines)
  |-> "Read retrieval-led, not training-led"
  |-> Compressed docs index
  |-> Where to find things

Layer 2: .context/ files  (auto-generated indices)
  |-> TAG_INDEX.md (keyword search)
  |-> PROTOCOL_SUMMARIES.md (protocol lookup)
  |-> project_state.md (current status)

Layer 3: .framework/ modules  (identity & laws)
  |-> Core_Identity.md (Laws #0-#4, RSI, Bionic Stack)
  |-> Output_Standards.md (response formatting, reasoning levels)

Layer 4: .agent/ runtime  (skills & workflows)
  |-> workflows/ (14 slash commands)
  |-> skills/protocols/ (200+ modular skill files)
  |-> scripts/ (130+ Python automation scripts)
  |-> PROMPT_LIBRARY.md (14 reusable meta-prompts)

Layer 5: docs/  (deep documentation)
  |-> ARCHITECTURE.md, SPEC_SHEET.md, etc.
```

### Key Design Principles

1. **Minimal Entry Point**: CLAUDE.md is ~80 lines. It doesn't try to contain everything. It tells the agent WHERE things are and HOW to find them.

2. **Retrieval-First Philosophy**: Instead of front-loading instructions, Athena teaches the agent to search. The TAG_INDEX enables keyword-based file discovery.

3. **On-Demand Loading**: Context is loaded only when needed:
   - User query about architecture -> load `System_Manifest.md`
   - Skill request -> load `SKILL_INDEX.md`
   - `/think` invoked -> load `Output_Standards.md`

4. **Triple-Lock Enforcement**: Every response must: Search -> Save -> Speak. This ensures the agent always checks memory before responding and persists any new insights.

5. **Active Injection over Passive Documentation**: Boot scripts (`boot_knowledge.py`) print critical constraints to stdout during `/start`, forcing them into the context window rather than hoping the agent will read them.

6. **Compressed Representations**: The docs index uses pipe-delimited format for token efficiency. Protocol summaries are tables, not full documents.

## 6. Comparison with Our CLAUDE.md

### Our CLAUDE.md (~300 lines)

| Aspect | Our Approach | Athena Approach |
|--------|-------------|-----------------|
| **Size** | ~300 lines, dense | ~80 lines, lean |
| **Philosophy** | Front-load all rules | Point to where rules live |
| **Entry point** | IS the rulebook | IS a directory/pointer |
| **Agent behavior** | Prescriptive ("ALWAYS do X, NEVER do Y") | Retrieval-led ("Read this file for context") |
| **Pipeline** | Embedded in CLAUDE.md (State Machine, Auto-Behaviors) | Separate workflow files (`.agent/workflows/`) |
| **Memory** | Embedded memory paths and update rules | Session lifecycle (`/start`, `/end`) handles memory |
| **Agent coordination** | Embedded team roles, skill mapping | Protocol 413 + COS agent system (separate files) |
| **Context loading** | Trigger tables embedded in CLAUDE.md | On-demand loading via TAG_INDEX + semantic search |
| **Token cost** | ~3000+ tokens always loaded | ~500 tokens entry + on-demand |
| **Model-agnostic** | Claude Code specific | Explicitly multi-model (routing table) |
| **Compaction safety** | Embedded recovery rules | Session persistence via `/start` + `/end` |

### Strengths of Athena's Approach

1. **Token Efficiency**: ~80 lines vs ~300 lines means the CLAUDE.md consumes ~6x fewer tokens on every context window load. Since CLAUDE.md is loaded for EVERY interaction, this adds up massively.

2. **Separation of Concerns**: Rules live in their own files, not crammed into one mega-document. Changes to one protocol don't require editing CLAUDE.md.

3. **Model Agnostic**: AGENTS.md works with any agent IDE. Our CLAUDE.md is tightly coupled to Claude Code conventions.

4. **Discoverable**: TAG_INDEX + semantic search means the agent can find relevant context dynamically, rather than having all rules pre-loaded.

5. **Scalable**: Athena scales to 308+ protocols without bloating the entry file. Our approach would require more lines for every new behavior.

### Strengths of Our Approach

1. **Explicit Compliance**: Every rule is right there -- the agent can't miss it. Athena relies on the agent choosing to search, which it might not always do.

2. **Compaction Safety**: Our embedded "after compaction" rules ensure the agent recovers correctly. Athena relies on the `/start` boot sequence for this.

3. **Pipeline Protocol**: Our embedded pipeline state machine is self-contained. Athena's pipeline equivalent is spread across multiple workflow files.

4. **Hard Constraints Table**: Our explicit "FORBIDDEN" and "HARD CONSTRAINTS" sections are unambiguous. Athena's constraints are distributed across Core_Identity.md and various protocols.

5. **No External Dependencies**: Our CLAUDE.md works standalone. Athena needs its full file tree to function.

### Synthesis: Best of Both

An ideal approach might combine:
- **Athena's lean entry point** (save tokens on every load)
- **Our explicit hard constraints** (never ambiguous)
- **Athena's retrieval-first philosophy** (scalable)
- **Our compaction safety rules** (critical for Claude Code)
- **Athena's TAG_INDEX concept** (keyword-based file discovery)
- **Our embedded pipeline** (self-contained state machine)

## 7. Novel Concepts Worth Adopting

### 7.1 Compressed Docs Index
Pipe-delimited file tree at top of CLAUDE.md -- gives the agent a mental map in ~200 tokens.

### 7.2 TAG_INDEX Auto-Generation
Script that indexes all files by tags/keywords, enabling keyword-based retrieval without semantic search.

### 7.3 Active Knowledge Injection
Boot scripts that print constraints to stdout, ensuring they're in the context window rather than relying on the agent to read files.

### 7.4 Protocol 413 (Multi-Agent Safety)
Concrete rules for concurrent agent work -- we have similar concepts but less formalized.

### 7.5 COS Agent Panel
Named, persona-based review agents (Architect, Guardian, Skeptic, etc.) -- more formalized than our ad-hoc agent spawning.

### 7.6 Triple-Lock (Search -> Save -> Speak)
Enforces memory usage on every response -- prevents the agent from "forgetting" to check context.

### 7.7 Trilateral Feedback Loop
Using different models to validate each other -- prevents single-model blind spots.

### 7.8 Prompt Library
14 reusable meta-prompts for common tasks (Pre-Mortem, MECE Breakdown, Code Review, etc.) stored in `.agent/PROMPT_LIBRARY.md`.

## 8. Summary

Athena's instruction approach is fundamentally **retrieval-oriented**: a minimal entry point that teaches the agent HOW to find information rather than WHAT all the rules are. This contrasts with our **prescriptive** approach where CLAUDE.md contains all behavioral rules directly.

Athena's approach scales better (308+ protocols without bloating the entry file) and is model-agnostic (AGENTS.md works across IDEs). Our approach provides stronger compliance guarantees (all rules are always in context) and better compaction safety.

The ideal evolution would be a **hybrid**: lean CLAUDE.md (~100 lines) with hard constraints + compressed file index, pointing to a well-organized directory structure for detailed protocols, with auto-generated indices for discovery.
