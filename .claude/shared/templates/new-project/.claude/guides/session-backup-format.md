# Session Backup Format (Pyramid)

> WHAT: Structured session backup before /clear — README summary + optional detail files.
> WHEN: Context at 90%+, before /clear, or before ending a long session.

---

## Directory Structure

```
.claude/backups/
  session-YYYYMMDD-HHMM/       <- unique per session
    README.md                   <- Level 1: always created
    changes.md                  <- Level 2: only if relevant
    decisions.md
    debugging.md
    research.md
    plan-progress.md
  latest-session/               <- COPY of latest session dir
    README.md
    ...
  latest-safety-net.md          <- auto-generated by pre-compact hook
  safety-net-*.md               <- timestamped safety nets (last 3)
```

---

## Level 1: README.md (ALWAYS created)

100-200 lines. Summary of entire session. Must be self-sufficient for context recovery.

### Template

```markdown
# Session Backup: YYYY-MM-DD HH:MM

**Branch:** [current branch]
**Task:** [what was being worked on]

## What Was Done
- [Concrete change 1: file, function, what changed]
- [Concrete change 2]
- ...

## Where Stopped
[Exact stopping point: file path, function name, step N of plan, etc.]

## Key Decisions
- [Decision 1]: [rationale]
- [Decision 2]: [rationale]

## Active Problems
- [Bug/blocker 1]: [status, what was tried]
- [Error encountered]: [root cause if known]

## Next Steps (ordered)
1. [Most important next action]
2. [Second priority]
3. ...

## Files Changed
- `path/to/file.py` — [what changed]
- `path/to/other.ts` — [what changed]

## Level 2 Detail Files
- [changes.md](./changes.md) — detailed code changes
- [decisions.md](./decisions.md) — architecture decisions
(only list files that actually exist in this backup)
```

---

## Level 2: Detail Files (ONLY when relevant)

Create Level 2 files only when they add significant value beyond what README.md covers.

### changes.md

**When:** 5+ files changed, complex refactoring, multi-step implementation.

```markdown
# Code Changes Detail

## [Component/Module Name]

### Files
- `path/to/file.py` — [what changed, why]
- `path/to/related.py` — [what changed, why]

### Rationale
[Why this approach was chosen, alternatives rejected]

### Impact
[What depends on this change, downstream effects]
```

### decisions.md

**When:** Architecture decision made, technology choice, pattern selection.

```markdown
# Session Decisions

## Decision: [Title]
**Context:** [problem being solved]
**Chosen:** [solution selected]
**Why:** [rationale, trade-offs considered]
**Impact:** [affected files, future work implications]
```

### debugging.md

**When:** Non-trivial debugging session, multiple hypotheses tested.

```markdown
# Debugging Session

## Problem
[Symptoms, error messages, reproduction steps]

## Investigation
1. [Hypothesis 1] — [result]
2. [Hypothesis 2] — [result]

## Root Cause
[What was actually wrong]

## Fix
[What was done to fix it]

## Prevention
[How to avoid this in future]
```

### research.md

**When:** External research, comparison of approaches, deep exploration.

```markdown
# Research Findings

## Question
[What was being researched]

## Sources
[Docs, code, experiments, URLs]

## Key Findings
- [Finding 1]
- [Finding 2]

## Recommendations
[What to do based on findings]
```

### plan-progress.md

**When:** Multi-step plan in progress, tracking completion across waves.

```markdown
# Plan Progress

## Plan: [name or file location]

## Completed Steps
- [x] Step 1 — [brief result]
- [x] Step 2 — [brief result]

## Current Step
- [ ] Step N — [what remains, where stuck]

## Remaining Steps
- [ ] Step N+1 — [description]
- [ ] Step N+2 — [description]
```

---

## Rules

1. **README.md is mandatory** — every backup MUST have it
2. **Level 2 files are optional** — only create when they add value beyond README
3. **Be concrete** — file paths, function names, line numbers; not vague descriptions
4. **100-200 lines for README** — enough to restore context, not entire transcript
5. **Link Level 2 from README** — always list existing detail files in README's "Level 2" section
6. **Timestamp in directory name** — format `session-YYYYMMDD-HHMM` (no seconds)
7. **Copy to latest-session/** — always overwrite latest-session/ with current session dir

---

## Creating a Backup (agent workflow)

```
1. Create directory: .claude/backups/session-YYYYMMDD-HHMM/
2. Write README.md (mandatory, 100-200 lines)
3. Write Level 2 files if relevant
4. Delete .claude/backups/latest-session/ if exists
5. Copy session dir to .claude/backups/latest-session/
6. Rotate: delete session dirs beyond last 5
7. Update .claude/memory/activeContext.md
8. Tell user: "Backup saved. Run /clear to continue."
```

---

## Rotation

Agent manages rotation after creating a new backup:

- Keep last **5** `session-YYYYMMDD-HHMM/` directories
- Delete older session directories
- `latest-session/` is always overwritten with the newest session
- Safety-net files: hook keeps last 3 `safety-net-*.md`

---

## Recovery Flow (next session)

```
1. Read .claude/memory/activeContext.md          <- inter-session bridge
2. Read .claude/backups/latest-session/README.md <- last session summary
3. IF more detail needed:
   -> Read Level 2 files from latest-session/
4. IF older context needed:
   -> Browse session-*/README.md directories
5. IF no pyramid backup exists:
   -> Read latest-safety-net.md (hook-generated fallback)
```

---

## Safety Net

The `pre-compact.sh` hook automatically writes `latest-safety-net.md` when compaction fires.
This is a simple flat backup of activeContext + git state.
It does NOT replace the pyramid backup — it's a fallback if the agent didn't create one before /clear.
