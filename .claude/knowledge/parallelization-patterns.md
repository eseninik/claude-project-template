# Parallelization Patterns

**Для автоматического обнаружения возможности параллелизации задач.**

База знаний типовых паттернов, которые указывают на независимые подзадачи.

---

## Как использовать

**Когда:** AUTO-CHECK вариант C (контекстный анализ задачи без явного плана)

**Алгоритм:**
1. Парсинг описания задачи
2. Поиск совпадений с паттернами ниже
3. Извлечение подзадач
4. Проверка независимости
5. Формирование рекомендации

---

### Database Migration

**Триггеры:**
- "миграция баз*", "обнов* схем*"
- "добав* колонк*", "изменение структур*"

**Подзадачи:**
1. Migration script (migrations/{timestamp}_description.sql)
2. Model updates (models/*.py)
3. Data migration script (scripts/migrate_data.py)
4. Rollback script (migrations/{timestamp}_rollback.sql)
5. Тесты миграции (tests/test_migration.py)

**Зависимости:**
- Wave 1: [1, 4] (forward и rollback migrations)
- Wave 2: [2] (модели зависят от схемы)
- Wave 3: [3] (data migration зависит от схемы и моделей)
- Wave 4: [5] (тесты)

**ВАЖНО:** Миграции БД часто ПОСЛЕДОВАТЕЛЬНЫЕ (каждая зависит от предыдущей)

## Категория: Рефакторинг

### Code Refactoring по файлам

**Триггеры:**
- "рефактор*", "улучш* структур*"
- "переименова*", "разбить на модули"

**Анализ:**
- Если упомянуты конкретные файлы → проверить пересечение
- Если файлы РАЗНЫЕ → параллельно
- Если ОДИН файл → последовательно (или Worktree Mode если независимые функции)

**Подзадачи:** Зависят от конкретики, нужен анализ

## Общие индикаторы параллелизации

### Множественные сущности

**Паттерн:** "для каждого", "все компоненты", "несколько модулей"

**Пример:**
```
"Обнови models/user.py, models/order.py, models/product.py"
```

**Анализ:**
- 3 разных файла → 3 независимые задачи → параллельно

---

### Списки в задаче

**Паттерн:** Нумерованный или маркированный список

**Пример:**
```
"Нужно:
1. Добавить логирование
2. Обновить документацию
3. Написать тесты"
```

**Анализ:**
- 3 пункта → проверить независимость
- Логирование (код) + Документация (docs) + Тесты (tests) → разные файлы → параллельно

---

### Связки "И также", "Кроме того"

**Паттерн:** "X, и также Y", "X, кроме того Z"

**Пример:**
```
"Добавь валидацию email, и также создай endpoint для проверки"
```

**Анализ:**
- Валидация (validators/) + Endpoint (api/) → разные файлы → параллельно

---

### Последовательные действия с независимыми результатами

**Паттерн:** "Сначала X, потом Y, затем Z"

**Пример:**
```
"Сначала создай модель, потом API, затем тесты"
```

**Анализ:**
- Модель (models/) + API (api/) + Тесты (tests/)
- API зависит от модели → Wave 1: [модель], Wave 2: [API], Wave 3: [тесты]

**НО:** Если формулировка "сначала... потом" - это ПОСЛЕДОВАТЕЛЬНОСТЬ, а не параллелизация
**ВАЖНО:** Проверить ЗАВИСИМОСТИ, а не только порядок слов

---

## Индикаторы ПОСЛЕДОВАТЕЛЬНОСТИ (НЕ параллелить)

### Зависимые изменения

**Паттерн:** "используя результат", "на основе", "после того как"

**Пример:**
```
"Создай функцию process_data(), используя её результат для generate_report()"
```

**Анализ:**
- generate_report ЗАВИСИТ от process_data → ПОСЛЕДОВАТЕЛЬНО

---

### Один файл, одна функция

**Паттерн:** Изменения в одной функции/классе

**Пример:**
```
"Добавь валидацию в функцию save_user() и логирование в ту же функцию"
```

**Анализ:**
- Один файл, одна функция → ПОСЛЕДОВАТЕЛЬНО (или Worktree Mode если уверены в независимости)

---

### Миграции с зависимостями

**Паттерн:** Database migrations, schema changes

**Пример:**
```
"Добавь таблицу users, затем таблицу orders с foreign key на users"
```

**Анализ:**
- orders зависит от users → ПОСЛЕДОВАТЕЛЬНО

---

## Алгоритм определения

### Шаг 1: Парсинг задачи

```
1. Извлечь ключевые слова
2. Проверить совпадения с паттернами выше
3. Если совпало → использовать подзадачи из паттерна
4. Если не совпало → извлечь подзадачи из структуры текста (списки, связки)
```

### Шаг 2: Проверка независимости

```
Для каждой пары подзадач (A, B):

IF разные файлы:
  → НЕЗАВИСИМЫЕ

ELSE IF один файл, разные функции/классы:
  → НЕЗАВИСИМЫЕ (Worktree Mode возможен)

ELSE IF один файл, одна функция:
  → ЗАВИСИМЫЕ (последовательно)

ELSE IF B использует результат A:
  → ЗАВИСИМЫЕ (A → B)

ELSE:
  → НЕУВЕРЕН (спросить пользователя)
```

### Шаг 3: Формирование waves

```
Wave 1: Все независимые задачи без зависимостей
Wave 2: Задачи зависящие только от Wave 1
Wave 3: Задачи зависящие от Wave 2
...
```

### Шаг 4: Рекомендация

```
IF количество задач < 3:
  → ПОСЛЕДОВАТЕЛЬНО (не нужна параллелизация)

IF количество задач >= 3 AND есть независимые:
  → ПРЕДЛОЖИТЬ ПАРАЛЛЕЛИЗАЦИЮ

IF все задачи зависимые (цепочка):
  → ПОСЛЕДОВАТЕЛЬНО
```

## Примеры использования

### Пример 1: Совпадение с паттерном

**Задача:**
```
"Добавь JWT аутентификацию"
```

**Анализ:**
1. Ключевое слово: "JWT", "аутентификац*"
2. Паттерн: Аутентификация / Authentication
3. Подзадачи из паттерна:
   - Middleware (middleware/auth.py)
   - Login endpoint (api/auth.py)
   - Register endpoint (api/auth.py)
   - Session storage (models/session.py)
   - Tests (tests/test_auth.py)
4. Независимость: Разные части/файлы → Wave 1: [1,2,3,4], Wave 2: [5]
5. Рекомендация: **ПАРАЛЛЕЛИЗАЦИЯ** (4 задачи в Wave 1)

---

### Пример 2: Извлечение из списка

**Задача:**
```
"Нужно:
1. Добавить логирование в все endpoints
2. Обновить README.md
3. Написать smoke tests"
```

**Анализ:**
1. Список → 3 пункта
2. Проверка независимости:
   - Логирование (код) vs README (docs) vs Tests (tests) → разные файлы
3. Waves: Wave 1: [1,2,3] (все независимые)
4. Рекомендация: **ПАРАЛЛЕЛИЗАЦИЯ** (3 задачи параллельно)

---

### Пример 3: Последовательность

**Задача:**
```
"Создай функцию fetch_data(), затем используя её результат создай generate_report()"
```

**Анализ:**
1. "используя её результат" → зависимость
2. generate_report ЗАВИСИТ от fetch_data
3. Waves: Wave 1: [fetch_data], Wave 2: [generate_report]
4. Рекомендация: **ПОСЛЕДОВАТЕЛЬНО** (только 2 задачи, вторая зависит от первой)

---

### Пример 4: Неопределённость

**Задача:**
```
"Улучши производительность приложения"
```

**Анализ:**
1. Паттерн: Performance Optimization
2. НО: не ясно какие конкретно оптимизации
3. Рекомендация: **СПРОСИТЬ ПОЛЬЗОВАТЕЛЯ**
   ```
   "Вижу что задача про оптимизацию производительности.

    Типовые подзадачи:
    1. Profiling для выявления bottlenecks
    2. Database optimization (indexes, queries)
    3. Caching layer
    4. Background jobs

    Какие именно оптимизации нужны? Или профилировать сначала?"
   ```

## Уверенность в рекомендации

### Высокая уверенность (>90%)

- Совпадение с паттерном из базы знаний
- Явный список с независимыми файлами
- Чёткие формулировки

**Действие:** Предложить параллелизацию уверенно

---

### Средняя уверенность (60-90%)

- Частичное совпадение с паттерном
- Неявные подзадачи, но разные файлы
- Нужны предположения о зависимостях

**Действие:** Предложить параллелизацию с пометкой "рекомендуется"

---

### Низкая уверенность (<60%)

- Нет совпадения с паттерном
- Неясно сколько подзадач
- Неясны зависимости

**Действие:** Спросить пользователя детали

---

## Интеграция с AUTO-CHECK

**Когда использовать эту базу:**

AUTO-CHECK вариант C:
```
C. Контекстный анализ задачи
   1. cat .claude/knowledge/parallelization-patterns.md
   2. Применить алгоритм определения
   3. IF уверенность > 80% AND 3+ задачи:
      → Предложить создать tasks/*.md
   4. ELSE IF уверенность > 60%:
      → Предложить варианты (parallel/sequential)
   5. ELSE:
      → Спросить детали
```

- Добавлять новые паттерны из реальных задач

- Добавлять domain-specific паттерны (Telegram bots, web scraping, etc.)

```markdown
### [Название паттерна]

**Триггеры:**
- "ключ* слов*", "фраз*"

**Подзадачи:**
1. Описание (file.py)
2. Описание (file2.py)

**Зависимости:**
- Wave 1: [список]
- Wave 2: [список]

**Файлы:**
- file.py
- file2.py
```
