# Git Workflow

## Назначение
Стратегия работы с git: ветки, коммиты, PR процесс.

---

## Структура веток

```
main        ← Продакшен, автодеплой
  └── dev   ← Рабочая ветка, сюда коммитим
       └── feature/xxx  ← Большие фичи (опционально)
```

**Основные ветки:**
- `main` - продакшен, только через merge из dev
- `dev` - основная рабочая ветка

**Feature ветки (опционально):**
- `feature/<name>` - для больших фич
- `fix/<name>` - для сложных фиксов

---

## Когда создавать feature ветку

**Работай в `dev` если:**
- Задача занимает < 1 дня
- Изменения локальные, не ломают другие части
- Один человек работает над проектом

**Создавай feature ветку если:**
- Задача большая (> 1 дня)
- Нужен code review
- Несколько человек работают параллельно
- Изменения рискованные

---

## Commit Messages

**Формат:**
```
type: краткое описание

[опционально: подробности]
```

**Types:**
- `feat:` - новая функциональность
- `fix:` - исправление бага
- `refactor:` - рефакторинг без изменения поведения
- `test:` - добавление/изменение тестов
- `docs:` - документация
- `chore:` - обслуживание (зависимости, конфиги)

**Примеры:**
```
feat: добавить команду /help
fix: исправить ошибку при пустом вводе
refactor: выделить UserService из handlers
test: добавить тесты для PaymentService
```

---

## Workflow коммитов

**Правило:** Коммить СРАЗУ после каждого завершенного изменения.

```bash
# После каждого изменения:
git add <измененные файлы>
git commit -m "type: описание"
git push origin dev
```

**НЕ накапливай** много изменений в одном коммите!

---

## Merge в main (деплой)

**Когда мержить:**
- Фича готова и протестирована
- Все тесты проходят
- Код проверен (self-review минимум)

**Как мержить:**
```bash
git checkout main
git merge dev
git push origin main
# GitHub Actions автоматически деплоит
```

---

## PR процесс (если используется)

**Создание PR:**
1. Запушить feature ветку
2. Создать PR в GitHub
3. Заполнить описание (что и зачем)
4. Запросить review

**Checklist перед merge:**
- [ ] Тесты проходят
- [ ] Линтер не ругается
- [ ] Код понятен
- [ ] Нет TODO в коде (или они задокументированы)

---

## Security Gates

**Перед каждым push:**
- [ ] Нет секретов в коде (токены, пароли)
- [ ] `.env` в `.gitignore`
- [ ] Sensitive данные не в логах

**Инструменты:**
- `gitleaks` - сканер секретов
- Pre-commit hooks
